# PaintPlay 引き継ぎドキュメント

年少さん（未就学児）向けの絵の具あそびアプリ。React単一コンポーネント（Claude Artifacts上で開発）。
このドキュメントは、別環境（ローカルAI／別チャット）に持ち込んで開発を継続するための引き継ぎ用。

---

## 1. ファイル

- 本体：`PaintPlay.jsx`（約1070行、単一ファイル、default export の React コンポーネント）
- 外部依存：`react` の `useState / useRef / useEffect / useCallback` のみ。追加ライブラリなし。
- Canvas 2D API を素で使用。Tailwind等は使わずインラインスタイル。

## 2. ターゲットユーザーと設計方針

- 年少さん（3〜4歳想定）が指タップ／指なぞりで遊ぶ想定。マウス操作は二の次。
- 誤操作防止（全消しは長押し必須）、色選択の迷子防止（パレット点滅ガイド）など、UI操作の「迷わせない」設計を優先している。
- 見た目は絵の具箱・スケッチブック・水入れなど実物メタファーで構成。フォントは丸ゴシック系指定。

## 3. アーキテクチャ概要

### 3.1 色・混色システム
- `PAINTS`：色ID→表示名・hex・枠線色の辞書（15色）。
- `TUBE_COLORS`：実際にチューブとして並ぶ6色（red/blue/yellow/white/black/lightBlue）。
- 混色ロジックは3段構え：
  1. `exactMixMap`：3色以上の特定組み合わせを厳密指定（例：白+赤+黄→クリーム）
  2. `pairMixMap`：2色ペアのテーブル（例：赤+青→紫）
  3. どちらにも無ければ、逐次2色混ぜでRGB平均→最も近い登録色にスナップ（`computeMix` 内フォールバック）

### 3.2 状態管理（すべて `PaintPlayApp` 内、Reduxなど不使用）
主要 state：
- `wells`：パレット5穴の配列。各穴は `{ colorId, ingredients }`
- `selectedWell`：選択中の穴インデックス
- `brushColor` / `brushSize`：筆の色・太さ（`thin/normal/thick`）
- `pendingColor`：チューブでタップしたが、まだパレットに入れていない色（**チューブ→パレット確定の2段階フロー**の要）
- `paletteBlinking`：パレット点滅トリガー
- `wetnessRef`（ref）＋ `wetnessUI`（state）：筆の「絵の具残量」0〜1。描くほど減り、パレットで取り直すと1に戻る
- `canUndo` / `undoStack`（ref、Canvasスナップショット最大10件）

### 3.3 操作フロー（★仕様のキモ）
```
チューブをタップ
  → pendingColor がセットされ、選択中パレット穴が点滅
  → パレット穴をタップして初めて色が確定して入る（誤タップ防止のワンクッション）
  → 筆にも自動で色がつく（wetness = 1 にリセット）
```
このフローは過去のやり取りで「チューブ即パレット反映」から明示的に変更されたもの。**元に戻さないこと。**

### 3.4 描画エンジン（今回のブラッシュアップの中心）
2つのサブシステムが独立して動いている：

**(A) 筆本体：スタンプ方式ブラシ**（147〜225行あたり）
- `makeBrushSprite(hex)`：色ごとに、中心が濃く外縁が透明になるラジアルグラデーションのスプライトを事前生成（`SPRITE_PX=64`）
- `initStrokeState(size)`：一筆ごとに呼ぶ。穂先の「毛」を4本ランダム生成（オフセット・濃さ・太さ・かすれやすさ）
- `strokeSegment()`：スプライトを進行方向に連打（stampAt）。速度で太さが変化（遅い→太い、速い→細い、`state.width` へ指数移動平均で追従＝筆のしなり表現）。かすれ時（wetness低）はスタンプの抜けが増える
- `strokeDot()`：タップ時（動かさず置いた時）の穂先タッチ表現
- `handlePointerUp` 内に「筆離れの払い（テーパー）」処理あり：最後の進行方向へ幅を1ステップごと0.58倍しながら5ステップ延ばして自然に抜ける

**(B) 滲み：ブロブ時間差拡散**（226〜262行、状態は `bleedsRef`）
- `makeBlobShape()`：9頂点の不定形閉曲線パラメータを生成（正円だと水彩に見えないための工夫）
- `spawnBleed(x, y, hex, size, strength)`：滲みを1個登録。`runBleedLoop` の rAF ループが `dur`（450〜850ms）かけて `r0→r1` にease-outで拡大しながら極薄アルファを毎フレーム加算塗り
- 呼び出し箇所：`pointerDown`（置いた瞬間）／`pointerMove`（一定距離ごとに確率発生）／`pointerUp`（筆を離した場所に2〜3個＝水彩の"バックラン"表現）
- **直近の修正**：滲みすぎ問題への対処として `r1` の倍率を `0.9〜2.0倍` → `0.55〜1.0倍` に縮小、`baseAlpha` も減量、`strength` に上限1.2を設定済み。ただし **この修正はまだユーザーによる実機確認前**（会話がここで途切れている）。

### 3.5 その他UI部品
- `PaintTube` / `PaletteWell`：選択中はスケール＆発光。`PaletteWell` は `blinkActive` 時に300ms周期で明滅（`useBlinkAnimation` フック）
- `BrushPanel`：筆先プレビュー（wetnessに応じて上から白でマスク＝残量表現）＋太さ3段階切替
- `WaterCup`：タップで筆をリセット（`wetnessRef=1`に戻る）。水の汚れ具合を8段階で色変化
- `Cloth`：選択中パレット穴のみクリア
- `UndoButton`：Canvasスナップショット方式（`bleedsRef` も同時にキャンセルしないと、undo後に古い滲みが継続描画されるので要注意）
- `ClearButton`：**長押し1200ms**必須（`requestAnimationFrame`でプログレスリング表示）。誤タップ防止のため即時クリアには絶対に変更しないこと（ユーザー要望で明示的にこの仕様になった）

### 3.6 Canvas解像度対応
- `setupCanvas()`：`devicePixelRatio`（上限2）に合わせて実ピクセルを確保し、CSSサイズは論理ピクセルで指定。リサイズ時は既存描画をスナップショット→拡大コピーで保持。
- `ResizeObserver` でラッパー要素の変化を監視。

## 4. 既知の課題／未確認事項（次のセッションでまず見るべきところ）

1. **滲み量の再調整が未検証**：直前の修正（r1縮小・alpha減量）をユーザーはまだ実機で見ていない。「これでもまだ多い／もう十分」の評価待ち。
2. **スマホでの道具箱レイアウト**：過去に一度「クリアボタンが画面外にはみ出る」問題が発生し、チューブ・パレット穴・各アイコンのサイズを縮小して対応済み。ただし機種・画面幅によっては再発の可能性あり。`flexWrap` を使っている箇所（下段のパレット＋ツール群）の挙動は要目視確認。
3. **`bleedsRef` と Undo の整合性**：Undo実行時に `cancelBleeds()` を呼んでいるか確認すること（アニメーション中の滲みが巻き戻り後も残ると不自然）。
4. **パフォーマンス**：滲み同時アクティブ数の上限は24件（元36件から削減）。低スペック端末での実機確認はまだ。
5. **筆エンジンの「マーカー感」問題は解消したという評価だったが**、滲みが強すぎたせいで印象が変わっている可能性がある。滲み調整後に改めて筆致そのものの評価をユーザーに聞くべき。

## 6. ローカル環境での再開に必要なこと

- Node.js + React環境（Vite等）を用意し、`PaintPlay.jsx` を `export default` のままコンポーネントとして読み込めばOK。
- 外部UIライブラリ・CSSフレームワーク不要（インラインstyleのみ）。
- `navigator.vibrate` はブラウザ・OS依存のため、対応していない環境では単に無視される（try-catch済み、エラーにはならない）。
- ローカルで動作確認する際は**必ずタッチデバイス相当のエミュレーション**（Chrome DevToolsのタッチシミュレーション等）で見ること。マウスでも `pointerdown/move/up` は動くが、太さの速度連動などは指の動きとマウスの動きで体感が変わる。
